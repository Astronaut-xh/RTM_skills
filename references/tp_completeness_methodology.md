# TP完备性保证方法论

## 一、问题根源：为什么TP数量不稳定

| 问题 | 原因 | 影响 |
|------|------|------|
| TP数量不稳定 | 基于启发式提取，无固定分解规则 | 每次运行可能产生不同数量的TP |
| 粒度不一致 | 无明确的粒度定义标准 | 有的FL分解为1个TP，有的分解为4个TP |
| 覆盖不完整 | 未建立覆盖率模型 | 可能遗漏边界条件、异常路径 |
| 无量化验收标准 | 缺少完备性检查清单 | 无法判断何时"足够" |

**根因**：自由发挥式提取 → 不可复现、不可量化。
**解法**：正交分解法 + 覆盖矩阵 + 验收检查清单。

---

## 二、正交分解法

### 2.1 核心思想

对每个Feature，按 **功能路径 × 参数组合 × 预期结果** 三维正交分解，确保无遗漏。

```
TP完备性 = Σ(FL覆盖率) × 场景覆盖率 × 条件覆盖率

其中：
- FL覆盖率：每个FL至少1个正向TP
- 场景覆盖率：正常 + 异常 + 边界 + 压力
- 条件覆盖率：参数组合覆盖
```

### 2.2 四类必分解场景

对每个FL，必须回答以下4个问题，每个"是"对应至少1个TP：

| 编号 | 场景类型 | 判定问题 | 示例 |
|------|----------|----------|------|
| S1 | **正向功能** | 该FL的主功能路径有几种参数组合？ | 命令类型×通道模式、读路径/写路径 |
| S2 | **边界条件** | 该FL涉及的信号/寄存器/计数器有无边界值？ | 地址上下界、帧长最小/最大、计数器归零/溢出 |
| S3 | **异常处理** | 该FL涉及哪些错误码或异常路径？ | 前置检查错误、执行期错误、协议违规 |
| S4 | **状态转换** | 该FL涉及的FSM有哪些状态和转移？ | 正常路径转移、异常跳转、复位恢复 |

### 2.3 六条固定分解规则（保证可复现性）

```
规则1: 每种命令/操作类型 × 每种配置模式 = 必须有至少1个正向TP
规则2: 每种错误/状态码 = 必须有至少1个TP验证其触发与返回
规则3: 每种复位/异常场景 = 必须有1个TP（空闲/工作中/各关键FSM状态）
规则4: 每个FSM状态 = 至少在1个TP中出现
规则5: 多错误/多条件并发 = 至少1个TP验证优先级或互斥关系
规则6: 错误恢复 = 至少1个TP验证错误后模块可正常接受新命令
```

**这些规则写死在生成逻辑中，不依赖人工判断，确保每次结果一致。**

## 三、TP数量估算方法

### 3.1 维度提取（从LRS文档中提取以下维度）

从LRS文档中提取以下维度，建立覆盖矩阵：

| 维度 | 提取来源 | 示例 |
|------|----------|------|
| **命令/操作类型** | LRS命令定义章节 | opcode列表、CSR/AHB分类 |
| **配置模式** | LRS配置/模式章节 | 通道宽度、工作模式、使能组合 |
| **错误码** | LRS异常/状态码章节 | 状态码表、优先级链 |
| **复位场景** | LRS时钟复位章节 | 空闲复位、各FSM状态下的复位 |
| **协议边界** | LRS接口/时序章节 | 帧中止、超时、截断 |
| **总线/接口** | LRS总线接口章节 | 读写路径、2-phase流水、固定参数 |
| **性能** | LRS性能需求章节 | 延时、频率、吞吐 |
| **FSM状态** | LRS架构/FSM定义章节 | 前端/后端/总线FSM所有状态及转移 |

### 3.2 最小TP数量公式

```
最小TP数 = 命令类型数 × 配置模式数       (正向功能)
         + 错误码总数                      (异常覆盖)
         + 复位场景数                       (复位覆盖)
         + 协议边界场景数                    (边界覆盖)
         + 并发/优先级验证数                  (并发覆盖)
         + 错误恢复验证数                     (恢复覆盖)
```

### 3.3 参数组合覆盖矩阵模板

| 参数 | 取值 | 组合数 |
|------|------|--------|
| 配置模式1 | {取值列表} | N |
| 配置模式2 | {取值列表} | M |
| 命令/操作类型 | {取值列表} | K |
| 地址/数据对齐 | {有效, 无效} | 2 |
| 使能/控制信号 | {开, 关} | 2 |
| 总线响应 | {正常, 错误} | 2 |
| 超时 | {正常, 超时} | 2 |

**理论组合数** = 各维度取值数之积
**实际需覆盖的关键组合** ≈ 理论组合数的10%-15%（按正交法筛选）

## 四、完备性验收检查清单

### 4.1 FL→TP覆盖检查

```
对每个FL必须验证：
□ 至少1个正向TP（主功能路径）
□ 至少1个边界/异常TP（若该FL涉及可出错场景）
□ 涉及的参数组合是否完整覆盖
□ 涉及的FSM状态转移是否覆盖
```

**警告阈值**：若某FL仅有1个TP，需评估是否因功能单一（如中断/DFX/低功耗等辅助项）而无需补充。

### 4.2 命令/操作覆盖检查

```
对每种命令/操作类型必须验证：
□ 每种配置模式下至少1个正向TP
□ 该命令特有的错误码至少1个TP
□ 该命令在使能/禁用条件下的行为
```

### 4.3 错误码覆盖检查

```
对LRS中定义的每种错误/状态码必须验证：
□ 至少1个TP触发该错误码
□ 返回值与LRS定义的编码一致
□ 错误码优先级（若存在优先级链）有TP验证
□ 前置错误与执行期错误的互斥关系有TP验证
```

### 4.4 FSM覆盖检查

```
对LRS中定义的每个FSM必须验证：
□ 每个状态至少在1个TP中出现（含编码双写）
□ 正常路径转移覆盖：全链路IDLE→...→IDLE
□ 异常路径转移覆盖：错误跳转、复位跳转
□ 自恢复机制覆盖：错误后自动回到IDLE
```

### 4.5 复位场景覆盖检查

```
必须覆盖的复位场景：
□ 空闲态复位（基本功能）
□ 工作中复位（各关键FSM状态下各1个）
  - 接收阶段复位
  - 发送阶段复位
  - 等待/执行阶段复位
□ 复位后模块可正常接受新命令
```

### 4.6 协议边界覆盖检查

```
必须覆盖的协议边界：
□ 协议中止/截断（若支持提前终止）
□ 最小帧/最短数据（边界下限）
□ 超时（若有时序约束）
□ 并发/重入（若模块不支持并发，验证拒绝行为）
```

## 五、执行流程

```
Step 1: 从LRS提取覆盖维度
  ├── 提取命令/操作类型列表
  ├── 提取配置模式列表
  ├── 提取错误码列表及优先级链
  ├── 提取FSM状态及转移图
  └── 提取复位/超时/协议边界条件

Step 2: 正交分解生成TP
  ├── 规则1: 命令×配置模式 → 正向TP
  ├── 规则2: 每种错误码 → 异常TP
  ├── 规则3: 每种复位场景 → 复位TP
  ├── 规则4: 每个FSM状态 → 状态转换TP
  ├── 规则5: 多错误并发 → 优先级TP
  └── 规则6: 错误恢复 → 恢复TP

Step 3: 完备性验收
  ├── FL→TP覆盖检查（每FL≥2或功能单一豁免）
  ├── 命令×配置覆盖检查
  ├── 错误码全覆盖检查
  ├── FSM全状态覆盖检查
  └── 复位/边界场景覆盖检查

Step 4: 补充缺失TP
  └── 对未通过验收的项补充TP，直到全部通过
```

## 六、TP质量自检标准

### 6.1 TP描述质量

每个TP的描述必须包含 **条件/激励/期望** 三段，且满足：

| 维度 | 禁止（笼统） | 要求（精简） |
|------|-------------|-------------|
| 信号引用 | "配置模块" | 引用关键信号名和值，如`test_mode_i=1 / lane_mode_i=16-bit` |
| 状态码 | "返回错误" | 状态码名+编码值，如`STS_ALIGN_ERR(0x20)` |
| LRS关联 | 无 | 引用具体LRS章节，如`参考LRS §4.8` |

**TP描述精简原则：**
- 不必精确到第几拍
- 包含关键信号/参数、命令类型、预期状态码、LRS章节引用即可
- 格式：`条件：xxx 激励：xxx 期望：xxx，参考LRS §xxx`

**示例：**
```
条件：test_mode_i=1，en_i=1，lane_mode_i=16-bit模式
激励：通过WR_CSR/RD_CSR访问VERSION(0x00)、CTRL(0x04)寄存器
期望：CSR读写正确，返回STS_OK(0x00)，参考LRS §4.6
```

### 6.2 数量稳定性保证

同一LRS + 同一DR_FL，多次运行生成的TP数量必须一致。

**保证机制**：六条固定分解规则（第二章2.3节），不依赖人工判断。

### 6.3 豁免规则

以下FL类型允许仅有1个TP，无需强制补充到2个：

| FL类型 | 豁免原因 |
|--------|----------|
| 中断（MVP不实现） | 功能单一，仅验证"无中断行为" |
| DFX | 辅助验证项，仅验证观测点可见 |
| 低功耗（设计目标） | 非强制规格，仅验证基本不翻转行为 |
| Memory map | 由SoC约束，模块侧仅验证可达性 |
| 不支持项 | 仅记录不支持行为，供后续版本回归 |
